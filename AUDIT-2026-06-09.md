Audited 2026-06-09 against commit `aa33c95` (164 commits, Jan 19 → May 30, 2026).
Method: full directory mapping, manual read of core server/client modules, `tsc` typecheck (passed), full test run (`npm test`), `npm audit`. No code was modified.

## 1. Executive Summary

**Overall health: B.** This is a far healthier codebase than most solo-built products: strict TypeScript that compiles clean, ~30 server test files covering real integration paths, fail-fast production config validation, hashed API keys with scopes, PKCE OAuth, signature-verified PayPal webhooks, and documented backup/restore/rollback procedures. The architecture is a sane modular monolith that fits its single-server SQLite deployment.

**Top 3 risks:** (1) The installed Clerk SDK — the app's primary auth layer — carries a known **critical** authorization-bypass advisory, plus 8 high-severity CVEs elsewhere in the production dependency tree; (2) there is **no CI whatsoever**, so nothing enforces that tests or typecheck pass before code reaches the production deploy scripts; (3) there is **no rate limiting** on any endpoint of a product that spends real money per request on Anthropic/OpenAI calls.

**Top 3 opportunities:** (1) a one-day CI + dependency-patch pass eliminates both top risks almost entirely; (2) the test suite is currently broken on Windows — your own dev machine — and a three-line fix restores it; (3) the UI's anime theming (NERV boot screens, scanlines, Rajdhani headings) is the cheapest, highest-visibility thing standing between this and a credible academic product (see §7).

## 2. Repo Map

**Purpose:** ScholarMark — a research/writing workspace. Users upload sources (PDF/TXT/images with OCR), get AI-generated annotations, organize evidence into projects, clip from the web via a Chrome extension, and draft/verify academic writing through chat. Paid tiers (free/pro/max) gate token budgets and storage. Production at `app.scholarmark.ai` (Hetzner, PM2, nginx).

**Stack:** TypeScript end-to-end. React 18 + Vite + Wouter + TanStack Query + Tailwind/shadcn (client); Express 4 + Drizzle ORM + better-sqlite3 (server); Clerk auth + custom JWT/API-key/MCP-token schemes; Anthropic (chat/writing/verify), OpenAI (embeddings/annotation pipeline/OCR vision), Gemini (humanizer). Vitest for tests.

**Maturity:** early-production SaaS with at least one paying-customer path (PayPal/Venmo live). Treat as production, calibrated to one server and a small user base.

| Area | What it is |
|---|---|
| `server/` (~30 modules) | Express composition root (`index.ts`), auth (`auth.ts`), feature route files, AI pipelines (`pipelineV2.ts`, `writingPipeline.ts`, `openai.ts`), crash-safe OCR queue, analytics, OAuth/MCP |
| `client/src/` | SPA: pages, components, shadcn `ui/`, React Query hooks, export libs (docx/pdf) |
| `shared/schema.ts` | Drizzle schema + Zod types shared by both sides (736 lines) |
| `chrome-extension/` | MV3 clipper: context menu, popup, API-key handoff via `/extension-auth` |
| `mcp-server/` | Standalone MCP resource server (port 5002), bearer passthrough to the API |
| `tests/` | 30 files: 27 server (heavy on integration), 2 client, 1 shared |
| `deploy/`, `scripts/` | PM2 config, backup timer/service, restore drill, smoke tests, sequential test runner |
| Root docs | `ARCHITECTURE.md`, `CODEBASE*.md` ×3, `TEST_STRATEGY.md`, six `TASK-*.md` planning files |

**Control flow:** `server/index.ts` → CORS/body parsing/Clerk middleware → `registerOAuthRoutes` → `registerAuthRoutes` → `registerRoutes` (which fans out to feature route files) → error handler → static/Vite. Every protected route uses `requireAuth`, which resolves API key → JWT → Clerk session in that order, with per-resource ownership checks in handlers.

**Surprises:** no root `README.md`; no `.github/` at all; an enormous `.claude/` agent/skill ecosystem (29 agents, 10 skills) that dwarfs most companies' tooling; light theme is "Darling in the Franxx"-inspired and dark theme is Evangelion NERV; `mcp-server/dist/` build artifacts are committed.

## 3. Audit Report

Severity key: **C**ritical / **H**igh / **M**edium / **L**ow. Each finding is labeled **[fact]** or **[judgment]**.

### Security

- **C1 — Clerk SDK has a known critical authorization-bypass advisory.** [fact] `npm audit` flags `@clerk/shared` (critical: "Middleware-based route protection bypass") and `@clerk/express@1.7.76` / `@clerk/clerk-react@5.61.3` (high: "authorization bypass when combining organization, billing, or reverification checks"). Clerk is the primary session auth (`server/auth.ts:437-452`). Mitigating context [judgment]: the app gates routes via per-route `requireAuth` + DB-backed ownership checks rather than Clerk middleware matchers, so the bypass surface is likely narrow here — but running a critical-advisory version of your auth SDK in production is not a defensible position. Fix: upgrade Clerk packages, re-run the auth integration tests.
- **H1 — 14 production-dependency vulnerabilities (1 critical, 8 high).** [fact] Besides Clerk: `drizzle-orm@0.39.3` (SQL injection via improperly escaped identifiers), `express`→`qs` (DoS), `ws@8.18.x` (uninitialized memory disclosure), `lodash` (prototype pollution), `path-to-regexp` (ReDoS), `js-cookie`. Most are fixable with `npm audit fix`; drizzle needs a deliberate minor-version bump.
- **H2 — No rate limiting anywhere.** [fact] No limiter middleware exists in `server/` (the only rate-limit code is outbound batching in `server/replit_integrations/batch/utils.ts`). Login-adjacent endpoints, API-key auth (`server/auth.ts:346-410`), and the AI endpoints that spend Anthropic/OpenAI money are all unthrottled. Token *budgets* (`hasTokenBudget`, `server/auth.ts:557`) cap monthly totals per user but nothing stops burst abuse, credential stuffing against `sk_sm_` keys, or one free-tier user hammering uploads/OCR.
- **M1 — Hardcoded infrastructure in CORS allowlist.** [fact] `server/index.ts:56` allows `http://89.167.10.34` (raw server IP, plain HTTP) and lines 32–37 bake `claude.ai`/`scholarmark.ai` origins into code. Consequence: config drift when the server moves, and an HTTP origin weakens the otherwise HTTPS-only posture enforced by `productionConfig.ts:88-90`.
- **M2 — API response bodies logged up to 2,000 chars.** [fact] `server/index.ts:116-134` serializes JSON responses into stdout logs. For a research tool, responses contain users' documents, annotations, and drafts — this puts user content/PII into PM2 log files with no redaction or rotation policy visible in the repo.
- Input validation, secrets hygiene, webhook verification, admin gating: **healthy.** No hardcoded secrets found; uploads are extension+MIME filtered with size caps (`uploadMiddleware.ts`); PayPal webhooks are signature-verified (`paypalBillingRoutes.ts:212-231`); admin routes deny-by-default when `ADMIN_USER_IDS` is unset (`analyticsRoutes.ts:147-164`); SQLite access goes through Drizzle/prepared statements throughout.

### Testing

- **H3 — The test suite cannot run on Windows (your dev machine).** [fact] `tests/server/helpers/bootstrapTempWorkspace.ts:17` calls `symlink()`, which requires admin/Developer Mode on Windows. `imageRoutes.test.ts` fails with `EPERM`, and because `scripts/run-vitest-sequential.mjs:20-22` exits on first failure, the remaining ~20 integration files (auth, oauth, uploads, web clips, writing — the crown jewels of the suite) never execute locally. Fix is one line: pass `"junction"` as the symlink type on win32.
- **M3 — Client coverage is near zero.** [fact] 2 client test files (`accountUtils`, `documentExportUtils`) vs. ~25 significant components/hooks. The most complex client logic — `useWritingChat.ts` (stream parsing, 19.8KB), `ProjectWorkspace.tsx`, export pipelines (`markdownToDocx.ts`, `pdfExport.ts`) — has no tests. [judgment] Server-first coverage was the right call; the streaming hooks are the next-highest-value target.
- Test quality otherwise: **healthy.** Integration tests assert real behavior (status codes, DB effects, auth rejection paths), not just execution; `vitest.config.ts` enforces `passWithNoTests: false`.

### Architecture & design

- **M4 — God files concentrate risk.** [fact] `server/chatRoutes.ts` 1,690 lines, `server/projectRoutes.ts` 1,540, `server/oauthRoutes.ts` 1,208, `client/src/pages/ProjectWorkspace.tsx` 1,477, `client/src/components/WritingChat.tsx` 1,269, `client/src/pages/ProjectDocument.tsx` 1,136. [judgment] These mix HTTP handling, prompt construction, streaming protocol, and business logic in single files; every feature touches them, so merge friction and regression risk compound. The module *boundaries* between files are otherwise clean (storage layers, pipelines, route registration are well separated).
- **M5 — Two competing schema-migration mechanisms.** [fact] `server/db.ts:40-243` runs `CREATE TABLE IF NOT EXISTS` + `ensureColumn()` ALTERs at import time, while `drizzle-kit push` (`package.json` `db:push`) manages the Drizzle-defined tables, plus a loose SQL file in `scripts/sql/`. [judgment] This works on one box but means no versioned, replayable migration history — schema state is "whatever ran". A drift between the raw-SQL tables and `shared/schema.ts` will be invisible until something breaks.
- Layering, coupling, data flow: **healthy** for the deployment model. The modular monolith + SQLite-on-one-box is the right architecture for this product stage; do not microservice this.

### Performance

- **M6 — Embedding search is a full-table scan in JS.** [fact] Embeddings are stored as JSON text columns (`shared/schema.ts:50,338,374,388,414`) and similarity is `cosineSimilarity()` in a loop (`server/openai.ts:84-96`, used by `projectSearch.ts`). Every search JSON-parses and dot-products every chunk. [judgment] Fine at current scale; becomes seconds-per-search somewhere around 10⁴–10⁵ chunks. Defer until felt, then adopt `sqlite-vec`.
- **M7 — Uploads fully buffered in RAM.** [fact] `uploadMiddleware.ts:37-86` implements memory storage with a 50MB/file cap. A handful of concurrent large uploads (plus OCR buffers, `sharp`, and PDF parsing) competes for memory on the single server. [judgment] Acceptable now; cap concurrent uploads or stream to disk if memory pressure appears.
- `maxWorkers: 1`, sequential test runner: a deliberate SQLite trade-off, fine.

### Dependencies

- Covered by C1/H1 above. Additional hygiene: **L1** [fact] `@types/cors` and `@types/multer` are in `dependencies` instead of `devDependencies` (`package.json:57-58`). **L2** [fact] `mcp-server/dist/` committed build artifacts; `.gitignore`'s `*.cjs` pattern contradicts committed `deploy/*.cjs` files (they must have been force-added).

### DevEx & operations

- **H4 — No CI.** [fact] No `.github/` directory; no workflow runs `tsc`, tests, or `npm audit`. The deploy path is manual shell scripts (`deploy/refresh-prod.sh`). Everything good about this repo's test culture is enforced by nothing but discipline.
- **M8 — No linting or formatting enforcement.** [fact] No ESLint or Prettier config exists anywhere in the project. Only `tsc` guards quality. Consequence: unused imports, inconsistent patterns, and `any`-creep are undetectable mechanically.
- **M9 — Ad-hoc logging, no error reporting.** [fact] 189 `console.*` calls across 23 server files; no structured logger, no Sentry/error-tracking integration. When production misbehaves, the only signal is PM2 stdout.
- Backups/restore/smoke: **strong.** `deploy/backup-data.sh` + systemd timer, `scripts/restore-drill.mjs` (a tested restore drill is rare and excellent), `smoke-prod.mjs` post-deploy checks.

### Documentation

- **M10 — Doc sprawl with unclear authority.** [fact] No root `README.md`. Three overlapping codebase docs (`CODEBASE.md`, `CODEBASE_INVENTORY.md`, `CODEBASE_REFERENCE.md` — 70KB), `ARCHITECTURE.md` ("last verified 2026-05-05"), six root `TASK-*.md` planning files, plus `.claude-docs/` (8 more). [judgment] For a newcomer (or future-you), there's no obvious entry point and stale copies will accumulate contradictions. One README + one maintained architecture doc beats seven partially-true ones.

### Code quality / dead code (Low)

- **L3** [fact] `eva-scanlines` is applied to the entire app (`client/src/App.tsx:79`) but defined nowhere in CSS — dead class. **L4** [fact] All shadow tokens have alpha `0.00` (`index.css:82-89`) — dead styling. **L5** [fact] `client/index.html:18` loads Rajdhani + Share Tech Mono from Google Fonts, while the CSS font stack (`index.css:74-76`) references Inter, Merriweather, and JetBrains Mono — **which are never loaded**; the app silently renders system fallbacks. **L6** [fact] `maximum-scale=1` in the viewport meta (`index.html:5`) blocks pinch-zoom — an accessibility problem for a reading app. **L7** [judgment] `BootSequence` (fake NERV boot animation) runs on every page load before content (`App.tsx:78`), delaying time-to-content for pure decoration.
- Error handling generally: **healthy.** No empty catch blocks found in `server/`; central error handler distinguishes Multer/URI/status errors (`index.ts:174-198`); SSE streams have heartbeats and sanitized errors (`sseUtils.ts`).

### Strengths (preserve these)

1. Strict TS, clean `tsc`, shared Zod+Drizzle schema as single source of truth.
2. Multi-scheme auth done carefully: hashed keys, scopes, PKCE, expiry, prod guards against test keys and `LOCAL_DEV_AUTH`.
3. `productionConfig.ts` fail-fast validation — with its own test file.
4. Crash-safe OCR job queue with checkpointed page results.
5. Real integration tests, restore drills, smoke tests, documented rollback.
6. Disciplined conventions: route registration pattern, storage modules, ownership-check helpers (`getOwnedConversationOr404`).

## 4. Improvement Strategy

**Theme A — Nothing enforces quality automatically.** (H3, H4, M8) Every guarantee lives in your habits. *Target:* a GitHub Actions workflow that runs typecheck + full test suite + `npm audit` on every push, with the suite runnable on both Windows and Linux. *Principle:* automate what you already do manually so it can't silently stop happening.

**Theme B — The security posture is good design on a stale foundation.** (C1, H1) The auth *architecture* is solid; the *packages* under it are vulnerable. *Target:* zero critical/high advisories in the production tree, re-checked automatically in CI weekly.

**Theme C — A paid AI product with no abuse valve.** (H2, M7) Per-user monthly budgets exist, but bursts, stolen keys, and free-tier hammering are uncapped, and the cost lands on your API bill. *Target:* per-IP and per-user rate limits on auth and AI/upload endpoints.

**Theme D — Six files own most future change-cost.** (M4) *Target:* no new logic added to the six god files; split server route files by feature on first touch; extract client components opportunistically. Not a big-bang refactor.

**Theme E — Production is a black box until a user complains.** (M2, M9) *Target:* structured logger with body redaction + a free-tier error tracker (Sentry).

**Explicitly NOT recommended:** migrating off SQLite (it fits this scale; you have backups and drills); a vector database (M6 is theoretical until corpora grow); client test coverage parity with server (target only the streaming hooks and export utils); rewriting the doc set (consolidate, don't rewrite); any infrastructure beyond the single Hetzner box.

**"Done" looks like:** CI green required to deploy · `npm audit --omit=dev` reports zero critical/high · `npm test` passes on Windows and Linux · rate limits return 429s in an integration test · server log lines are JSON with no response bodies · root README exists and links to one authoritative architecture doc · zero Critical findings on re-audit.

## 7. Academic UI & Typography Plan

**Where it stands:** light mode is "Darling" (muted rose `#D4556B` primary), dark mode is "Eva/NERV" (orange-on-purple, Rajdhani display headings), with a fake boot sequence, a data ticker, and a scanlines class on the app shell. Meanwhile the CSS *declares* a serious stack — Inter / Merriweather / JetBrains Mono (`index.css:74-76`) — **that is never loaded** (`index.html:18` loads only Rajdhani + Share Tech Mono), so body text is actually system-default. The radius tokens are already small and flat (good — academic UIs are quiet), and `@tailwindcss/typography` is already installed.

**Design principle:** a research tool earns trust by looking like a *well-set book with quiet instruments around it*. Serif for reading surfaces, neutral sans for chrome, one restrained accent, generous measure and line-height, zero decoration that moves.

### Step 1 — Typography (S effort, biggest visible change)

- **Reading face (documents, writing pane, chat prose):** *Source Serif 4* or *Literata* (both variable fonts designed for long-form screen reading; Literata is the Google Play Books face). Swap into `--font-serif`.
- **UI face (nav, buttons, tables, sidebar):** keep *Inter* — it's already declared; just actually load it. (Alternative with more warmth: *Public Sans* or *IBM Plex Sans*.)
- **Mono (citations metadata, code, IDs):** *JetBrains Mono* as declared, or drop to system mono to save a request.
- **Self-host via `@fontsource-variable/*` packages** instead of Google Fonts CDN: faster (no third-party round trip), works offline, and avoids the Google Fonts GDPR problem — which academics notice.
- **Reading defaults:** document viewer and writing pane get `font-serif`, 17–18px, `line-height: 1.7`, `max-width: 70ch`, `hyphens: auto`, `text-rendering: optimizeLegibility`, and `font-variant-numeric: oldstyle-nums` for in-prose numbers.
- **Headings:** same serif at weight 600 with tight tracking — *delete the Rajdhani dark-mode heading rule* (`index.css:237-241`).

### Step 2 — Color (S effort)

Keep the warm-paper light background (`#FAFAF8` is already right). Replace the anime accents:

| Token | Now | Proposed |
|---|---|---|
| `--primary` (light) | rose `#D4556B` | Oxford blue `hsl(215 45% 32%)` — the color of a university press spine |
| `--secondary` | Franxx blue | slate `hsl(215 15% 45%)` |
| highlight/annotation hues | chart rainbow | scholarly markers: ochre `hsl(40 65% 55%)`, sage `hsl(150 25% 45%)`, burgundy `hsl(350 45% 40%)`, slate blue — muted, print-like |
| dark `--background` | NERV purple `hsl(260 30% 8%)` | neutral warm-black `hsl(220 10% 10%)` ("library at night") |
| dark `--primary` | Eva orange | desaturated gold `hsl(40 50% 60%)` or keep a calmer blue |

Run every pair through a contrast checker (≥4.5:1 body, ≥3:1 large text).

### Step 3 — Strip the sci-fi chrome (S effort)

Remove `BootSequence` (it delays first paint for decoration), `DataTicker`, the dead `eva-scanlines` class (`App.tsx:79`), the `eva-*` Tailwind tokens (`tailwind.config.ts:84-92`), and the `Share Tech Mono`/`Rajdhani` font requests. If the theme is dear to you, hide it behind a `?theme=eva` easter egg rather than the default.

### Step 4 — Academic affordances (M effort)

- Small-caps, letter-spaced section labels (`font-variant-caps: all-small-caps; letter-spacing: 0.08em`) for sidebar/group headers — replaces the current mono-uppercase "Loading View" style.

- Customize the `prose` plugin theme once (in `tailwind.config.ts`) so chat output, document summaries, and the writing pane all inherit the same book-like rhythm instead of styling each separately.
- Tables (analytics, citations): rules above/below only, no zebra striping — booktabs style.

### Step 5 — Verify (S)

Screenshot before/after of: document reader, writing pane, chat, project workspace. Check dark mode contrast. Total estimated effort: **1–2 days** for steps 1–3 (the 80% win), another day for step 4.

## 8. "Fable Writing Bench" — a benchmark to understand Fable 5's writing capability

**Goal:** not a generic eval — a benchmark that tells *you* whether Fable-class models can do the writing work ScholarMark's pipeline demands, and where they fail. It plugs into the repo you already have: `.claude/skills/scholarmark-thesis/benchmarks/suite.template.json` and `scripts/verify_quotes.py` / `quote_gate.py` give you the harness bones.

### Design: 6 task families × hybrid scoring

| # | Family | Task | Primary scoring (deterministic) | Secondary (LLM judge) |
|---|---|---|---|---|
| 1 | **Evidence-grounded synthesis** | Given 4–6 source excerpts, write an 800-word argument citing ≥5 quotes | **Quote fidelity rate**: every quoted span must appear verbatim in a source (string match after whitespace normalization — exactly what `verify_quotes.py` does). Also: fabricated-citation count (must be 0) | argument quality rubric 1–5 |
| 2 | **Thesis construction** | Given a topic + evidence pool, produce 3 candidate theses + defend one | checklist: debatable? specific? scoped? (binary per criterion, judged) | counterargument-handling rubric |
| 3 | **Voice matching** | Given 2 samples of a target writer (~500 words), continue a third piece | stylometrics vs. target: mean sentence length ±15%, type-token ratio ±10%, comma/semicolon rate; embedding cosine to target > cosine to generic baseline | blind A/B: judge picks which of two continuations matches the samples |
| 4 | **Constrained revision** | Edit a flawed draft: fix passive voice + wordiness, *without altering quotes or citations* | diff check: quote spans byte-identical; passive-voice count reduced ≥80% (spaCy or regex heuristic); length reduced 10–25% | meaning-preservation rubric |
| 5 | **Long-form coherence** | 2,500–3,000-word structured paper from a fixed outline | outline adherence (every section header present, in order); repetition score (max 8-gram overlap between sections < threshold); word count ±10% | transition quality, no-contradiction check |
| 6 | **Citation mechanics** | Generate APA-7 / MLA-9 / Chicago footnote+bib from 10 metadata records (books, journals, websites, edge cases: no author, 12 authors, DOI vs URL) | **exact string match** against hand-verified answers — fully automatic | none needed |

### Anti-slop instrument (cuts across all families)

A banned-tell list scored on every output: *delve, tapestry, testament to, it's important to note, in conclusion* (when unprompted), *moreover*-chains, em-dash density >1.5/100 words, hedging stacks ("could potentially perhaps"). Report **tells per 1,000 words**. This is the single best cheap proxy for "does it read like AI."

### Protocol (rigor without overkill)

1. **N=3 samples per prompt** at default temperature; report median and range — single-shot results are noise.
2. **Judge ≠ subject**: score Fable outputs with a *different* model family as judge (e.g., GPT-class) and vice versa, to dodge self-preference bias; spot-check 10% by hand — you, reading them.
3. **Baselines**: run the identical suite on Sonnet 4.6 and Opus 4.8. "Fable scores 4.2" is meaningless; "Fable 4.2 vs Sonnet 3.6 on voice-match, parity on citations" is a decision.
4. **Frozen inputs**: pin source excerpts, outlines, and metadata records in the suite JSON so runs are comparable across months/model versions.
5. **Output**: one radar chart (6 axes = 6 families) per model + a table of deterministic metrics. The deterministic metrics (quote fidelity, citation exact-match, tells/1k words) are your headline numbers because they can't be judge-gamed.

### Suite skeleton (drop into `benchmarks/`)

```json
{
  "suite": "fable-writing-bench-v1",
  "models": ["claude-fable-5", "claude-opus-4-8", "claude-sonnet-4-6"],
  "samples_per_prompt": 3,
  "judge_model": "non-anthropic-judge",
  "tasks": [
    { "id": "synth-01", "family": "evidence_synthesis",
      "inputs": { "sources": ["fixtures/sources/industrial-rev-*.txt"], "length_words": 800, "min_quotes": 5 },
      "deterministic": ["quote_fidelity", "fabricated_citations", "word_count_tolerance:0.1", "tells_per_1k"],
      "judged": ["argument_quality"] },
    { "id": "cite-01", "family": "citation_mechanics",
      "inputs": { "records": "fixtures/citations/records.json", "styles": ["apa7", "mla9", "chicago-nb"] },
      "deterministic": ["exact_match_rate"] }
  ]
}
```

**Build order:** family 6 first (fully automatic, one afternoon, immediately tells you something real), then 1 (reuses `verify_quotes.py`), then the anti-slop scorer (one regex pass), then 3/4/5. Total: ~2–3 days to a v1 that produces honest numbers.
